Trả lời
1 Phân rã thành token Văn bản → các mảnh nhỏ (subword)

Bạn gõ: "Xin chào, hôm nay trời đẹp."

Xinchào,hômnaytrờiđẹp.
148262281411983456972163313
Chú ý "chào" bị tách thành 2 mảnh ("ch" + "ào") — đây là lý do văn bản tiếng Việt có dấu thường tốn token hơn so với văn bản tiếng Anh có cùng độ dài, vì bộ tách từ (tokenizer) chủ yếu học trên dữ liệu đa ngôn ngữ.
2 Token biến thành vector (embedding) Mỗi token ID → một dãy số nhiều chiều

Mỗi token ID được tra trong một "bảng embedding" khổng lồ để đổi thành vector:

ID 1482→ -0.221.570.98...0.31-1.02 (7680 chiều)
Các từ gần nghĩa nằm gần nhau trong không gian vector — "chó/dog" tạo thành 1 cụm, "mèo/cat" tạo cụm khác, còn "xe hơi" nằm tách biệt hẳn.
3 Transformer đọc toàn bộ context (Attention) Model "chú ý" đến phần liên quan nhất

Với câu hỏi hiện tại là "Tôi làm nghề gì?", model đọc lại cả lịch sử hội thoại và tính mức độ liên quan của từng câu:

"Tôi tên Bình."
"Tôi sống ở Hà Nội."
"Tôi thích AI."
"Tôi làm nghề gì?"
Câu "Tôi thích AI." nhận trọng số chú ý cao hơn hẳn "Tôi sống ở Hà Nội." vì liên quan trực tiếp đến câu hỏi. Đây cũng là lý do lịch sử hội thoại càng dài, model càng phải "quét" qua càng nhiều token để tìm phần liên quan — càng tốn token và chậm hơn.

5 Sinh token mới liên tục Lặp lại quy trình cho đến khi dừng
Con→mèo→đang→ngủ→...→trên→ghế→.
Token vừa chọn được nối vào cuối chuỗi, rồi bước 3 và 4 (transformer → dự đoán) lặp lại từ đầu để sinh token kế tiếp — cứ như vậy cho đến khi gặp tín hiệu dừng, "stop sequence", hoặc chạm max_tokens.

Nguồn tiêu hao

Vì sao AI "tốn" token?

Token bị tiêu theo tổng lượng context — đầu vào, lịch sử, tài liệu, và cả đầu ra.

💬
Context quá dài
Cuộc hội thoại càng nhiều tin nhắn, model càng phải đọc lại toàn bộ lịch sử mỗi lần trả lời.
📄
Tài liệu / nhiều file
PDF, code, log... đều bị đổi thành token. File càng lớn, càng nhiều file → càng tốn.
🧠
Suy luận sâu (reasoning)
Yêu cầu phân tích, so sánh phức tạp khiến model phải "suy nghĩ" qua nhiều bước trung gian.
📝
Output dài
Trả lời càng chi tiết, càng dài → phần sinh ra (completion) cũng tính vào tổng token.
🖼️
Hình ảnh
Ảnh độ phân giải cao phải mã hoá thành rất nhiều token trước khi model "nhìn" được.
💻
Code
Code có nhiều ký tự đặc biệt, thụt lề, ký hiệu → thường tốn nhiều token hơn văn bản thường.
Ước lượng nhanh

Công thức tính token

Không cần chính xác tuyệt đối — chỉ cần hiểu "tổng" này được cộng dồn từ đâu.

System Prompt+ Lịch sử hội thoại+ Tài liệu / File / Ảnh+ Câu hỏi hiện tại+ Phản hồi= Tổng số token
25.000
15.000
System prompt: 800 Lịch sử: 25.000 Tài liệu/File: 15.000 Câu hỏi: 300 Phản hồi: 2.000
800 + 25.000 + 15.000 + 300 + 2.000 = 43.100 token
⚠️
Context window có giới hạn (ví dụ 128K token cho một số model phổ biến). Nếu tổng vượt quá giới hạn này, hệ thống phải cắt bớt phần cũ hoặc từ chối xử lý — xem chi tiết ở phần tiếp theo.
Bộ nhớ làm việc

Context window là gì?

Mỗi model có giới hạn số token tối đa có thể xử lý trong một lần gọi.

Hệ thống
Lịch sử hội thoại
Tài liệu/File/Ảnh
Câu hỏi
Phản hồi
Hệ thống / hướng dẫn (system prompt) Lịch sử hội thoại Tài liệu / file / hình ảnh bạn cung cấp Câu hỏi hiện tại Phần trả lời đang được sinh
Tổng ≤ giới hạn (ví dụ: 128.000 token)
⚠️
Nếu vượt quá giới hạn → hệ thống phải tự động cắt bớt phần cũ trong lịch sử, hoặc từ chối xử lý yêu cầu của bạn cho đến khi context được rút gọn lại.

8 cách tiết kiệm token hiệu quả

So sánh cách làm thường gặp (dễ tốn token) với cách làm tiết kiệm hơn.

1. Đừng chat quá dài trong 1 thread
Rất nhiều
Không nênNhắn hàng trăm tin nhắn trong cùng 1 thread — context ngày càng dài.
Nên làmTách theo dự án: mỗi thread riêng cho Auth, Frontend, Database, Deployment...
2. Tóm tắt (summary) rồi tạo chat mới
Rất nhiều
Không nênĐể AI phải đọc lại toàn bộ lịch sử mỗi lần trả lời.
Nên làmTóm tắt ý chính (1, 2, 3...) rồi mở chat mới, tiếp tục từ bản tóm tắt.
3. Chỉ gửi phần cần thiết
Nhiều
Không nênGửi cả project: 500 file, 50.000 dòng code, 200 trang PDF.
Nên làmChỉ gửi file, đoạn code, hoặc phần nội dung thực sự liên quan.
4. Chia nhỏ tài liệu / code / nhiệm vụ
Nhiều
Không nênĐưa nguyên PDF 300 trang cho AI xử lý cùng lúc.
Nên làmChia thành từng phần (1–50, 51–100...) và xử lý lần lượt.
5. Dùng RAG / Knowledge Base
Rất nhiều
Không nênĐưa toàn bộ kho dữ liệu cho AI tự đọc hết mỗi lần hỏi.
Nên làmTìm và chỉ đưa 3–5 tài liệu liên quan nhất đến câu hỏi.
6. Yêu cầu trả lời ngắn gọn, có cấu trúc
Nhiều
Không nênĐể AI viết chi tiết, dài dòng mọi khía cạnh.
Nên làmYêu cầu trả lời theo bullet / tóm tắt / ví dụ / kết luận.
7. Tắt/giới hạn chế độ suy luận nặng
Trung bình – nhiều
Không nênLuôn bật chế độ reasoning sâu kể cả cho câu hỏi đơn giản.
Nên làmDùng chế độ thông thường, chỉ bật reasoning sâu khi thực sự cần.
8. Quản lý file & hình ảnh thông minh
Nhiều
Không nênUpload nhiều file, ảnh độ phân giải cao không cần thiết.
Nên làmChỉ upload đúng file/ảnh cần, nén nhẹ nếu được.
Nhớ nhanh

Mẹo áp dụng ngay hôm nay

  • Luôn kiểm tra số token đang dùng, nếu công cụ của bạn có hiển thị.
  • Xoá bớt phần lịch sử không còn cần thiết trong cuộc hội thoại.
  • Dùng từ ngữ ngắn gọn, rõ ràng khi đặt câu hỏi.
  • Ưu tiên bullet, bảng, code block thay vì mô tả dài dòng.
Quy trình gợi ý

Ghi lại tóm tắt khi code cùng AI (VS Code / Claude Code)

Thay vì để AI đọc lại toàn bộ lịch sử, hãy để nó đọc bản tóm tắt gọn gàng của chính nó.

1
Hoàn thành 1 nhiệm vụ
Bạn yêu cầu AI thực hiện 1 việc cụ thể (viết code, phân tích, viết tài liệu...) → AI trả về kết quả.
2
Tạo tóm tắt
Yêu cầu AI tự tóm tắt ngắn gọn: mục tiêu, những gì đã làm, các thay đổi chính, file liên quan, lưu ý quan trọng.
3
Lưu vào file Markdown
Lưu bản tóm tắt vào một file .md ngay trong dự án, đặt tên rõ theo ngày/nhiệm vụ.
4
Dùng làm context cho lần sau
Khi bắt đầu nhiệm vụ mới, chỉ cần đưa các file tóm tắt này — thay vì toàn bộ lịch sử hội thoại.
5
Lặp lại thành vòng lặp hiệu quả
Yêu cầu → AI trả lời → Tóm tắt → Lưu file → quay lại yêu cầu tiếp theo. Cứ thế cho mỗi nhiệm vụ mới.

Tóm lại

Nếu chỉ nhớ một điều mỗi mục

  • AI không hiểu chữ như con người — nó chỉ xử lý một chuỗi token (số).
  • Token bị tính trên tổng context: system prompt + lịch sử + tài liệu + câu hỏi + phản hồi.
  • Bạn càng cung cấp nhiều thông tin, hội thoại càng dài, file càng lớn → càng tốn nhiều token.
  • Mỗi model có context window giới hạn — vượt quá phải cắt bớt hoặc bị từ chối xử lý.
  • Giảm context không cần thiết (tóm tắt, chia nhỏ, chỉ gửi phần liên quan) là cách tiết kiệm token tốt nhất.
  • Ít token hơn đồng nghĩa với nhanh hơn, rẻ hơn, hiệu quả hơn.
