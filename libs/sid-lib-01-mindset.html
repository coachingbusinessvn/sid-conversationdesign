Định hướng 01

Strategic Thinking

Nhìn mục tiêu lớn, trade-off, thứ tự ưu tiên. Tránh làm chatbot "biết nhiều nhưng vô dụng" bằng cách xác định business outcome rõ trước khi chạm vào prompt.

Giải thích
Người làm non tay sẽ nhảy thẳng vào prompt khi nghe "xây chatbot CSKH". Người có strategic thinking hỏi: Bot để giảm ticket hay tăng conversion? KPI là gì? 80% câu hỏi thuộc nhóm nào? Scope của nó đến đâu?
Prompt mẫu
Trước khi xây chatbot này, hãy giúp tôi xác định: 1. Kết quả kinh doanh cụ thể muốn đạt? 2. Top 3 use case chiếm 80% nhu cầu? 3. Ranh giới scope: cái gì bot KHÔNG nên làm? 4. Kiến trúc phù hợp: prompt-only, RAG, tool-calling?
Use cases
Chatbot CSKH Product roadmap Chọn kiến trúc AI Scope MVP
Mô hình hóa 02

Systems Thinking

Nhìn chatbot như một pipeline nhiều tầng: input → retrieval → memory → tool → output → eval. Lỗi ở tầng nào gây vấn đề gì? Dependency loop ở đâu?

Giải thích
Chatbot thật có: instruction layer → retrieval layer → memory layer → tool layer → safety layer → eval layer. Nếu chỉ "vibe prompt" mà không nghĩ hệ thống: docs lộn xộn, retrieval lạc chủ đề, assistant nhớ nhầm, tool gọi sai lúc.
Prompt mẫu
Vẽ sơ đồ pipeline đầy đủ cho chatbot [X]: - Mỗi tầng làm gì? - Nếu tầng retrieval sai → output bị ảnh hưởng thế nào? - Failure points nào nguy hiểm nhất? - Feedback loop nào cần thiết kế?
Use cases
Debug chatbot Thiết kế multi-agent RAG architecture Production readiness
Mô hình hóa 03

Structural Thinking

Tổ chức thông tin thành schema, tầng, block, field, state. NLP mạnh nhất khi bạn nhìn thấy cấu trúc tri thức ẩn trong ngôn ngữ.

Giải thích
Thay vì prompt "hãy trả lời hay về bảo hiểm", hãy thiết kế: nhận diện intent → trích entity (loại BH, độ tuổi, thời hạn) → phân biệt fact vs recommendation → nếu thiếu dữ liệu thì hỏi lại → output theo 4 block: tóm tắt / điều kiện / ngoại lệ / bước tiếp theo.
Prompt mẫu
Hãy thiết kế schema cho câu hỏi người dùng trong domain [X]: - Intent taxonomy (phân loại ý định) - Entity cần trích xuất - Trạng thái hội thoại (state machine) - Output format JSON cho từng loại intent
Use cases
Knowledge base design NLP extraction Custom GPT Ontology
Mô hình hóa 04

Context Engineering

Không chỉ viết prompt — mà thiết kế toàn bộ ngữ cảnh AI nhận được: system prompt, memory, retrieved docs, tool results, conversation history. Chất lượng context = chất lượng output.

Giải thích
Context window là căn phòng AI tư duy. Context engineering là việc quyết định: thứ gì được vào phòng, xếp theo thứ tự nào, loại bỏ cái gì để tránh nhiễu. Prompt chỉ là một phần nhỏ trong bức tranh đó.
Prompt mẫu
Audit context window của tôi cho task [X]: - System prompt có rõ role + constraints chưa? - Retrieved chunks có relevant không, có nhiễu không? - Memory có cần truncate không? - Thứ tự thông tin có tối ưu chưa? (recency bias)
Use cases
Long-form AI agent RAG tuning Memory management Multi-turn chat
Xử lý vấn đề 05

Task Decomposition

Chia bài toán lớn thành các sub-task nhỏ, độc lập, có thứ tự. AI làm tốt nhất khi từng bước được định nghĩa rõ — không phải một prompt khổng lồ.

Giải thích
Thay vì "viết báo cáo thị trường hoàn chỉnh", hãy tách: (1) xác định scope → (2) thu thập data → (3) phân tích competitor → (4) synthesis insights → (5) viết executive summary. Mỗi bước là một lần gọi AI riêng biệt với output rõ ràng.
Prompt mẫu
Tách task [X] thành các sub-task nhỏ nhất có thể: - Mỗi sub-task: input là gì? output là gì? - Sub-task nào phụ thuộc vào nhau? - Sub-task nào có thể song song? - Checkpoint quality ở bước nào?
Use cases
Agentic workflow Chain-of-thought Complex research Code generation
Xử lý vấn đề 06

Logical Thinking

Thiết kế policy, decision rules, if/else trong hội thoại. Không có logic flow rõ, bot sẽ trả lời trôi chảy nhưng không đáng tin — đặc biệt trong domain chuyên môn.

Giải thích
Ví dụ chatbot tuyển dụng: nếu user hỏi JD → dùng knowledge base. Nếu hỏi tình trạng hồ sơ → yêu cầu mã ứng viên. Nếu thiếu mã → hỏi lại. Nếu không chắc → nói rõ chưa xác minh. Nếu liên quan thông tin cá nhân → từ chối suy đoán.
Prompt mẫu
Viết decision tree đầy đủ cho chatbot [X]: - Khi user hỏi [A] → bot làm gì? - Điều kiện nào kích hoạt tool gọi? - Khi nào bot phải hỏi lại? - Khi nào từ chối, khi nào escalate?
Use cases
Policy design Guardrails Instruction hierarchy Safety rules
Xử lý vấn đề 07

OODA Loop

Observe → Orient → Decide → Act. Vòng lặp ra quyết định nhanh của không quân Mỹ. Áp dụng vào AI: đừng để AI phản xạ ngay — ép nó qua 4 bước trước khi output.

Giải thích
Observe: AI quan sát input đầy đủ (không chỉ bề mặt). Orient: AI định hướng bối cảnh — lịch sử, vai trò, constraint. Decide: AI chọn strategy phù hợp từ nhiều options. Act: AI hành động với output đã được filter qua 3 bước trên.
Prompt mẫu
Trước khi trả lời, hãy đi qua OODA loop: OBSERVE: Tôi đang thực sự hỏi gì? Có thông tin ẩn nào không? ORIENT: Bối cảnh liên quan là gì? Constraint nào quan trọng? DECIDE: Có bao nhiêu cách tiếp cận? Cái nào tối ưu nhất? ACT: Bây giờ hãy đưa ra câu trả lời.
Use cases
High-stakes decisions Ambiguous requests Crisis response Agent reasoning
Kiểm soát CL 08

Feynman Technique

Yêu cầu AI giải thích 3 lớp: đơn giản như cho trẻ 12 tuổi → ngôn ngữ chuyên môn → chỉ rõ điểm mà bản thân AI không chắc. Ba lớp này lộ ra đâu là kiến thức thật.

Giải thích
Richard Feynman: nếu bạn không thể giải thích đơn giản, bạn chưa thực sự hiểu. Áp dụng vào AI: ép AI qua 3 lớp giải thích buộc nó phân biệt kiến thức thật với kiến thức "nghe có vẻ đúng" — đặc biệt quan trọng trong domain chuyên môn cao.
Prompt mẫu
Giải thích [khái niệm X] theo 3 lớp: LỚP 1: Như thể tôi 12 tuổi (không dùng jargon) LỚP 2: Ngôn ngữ chuyên môn đầy đủ LỚP 3: Những điểm nào bạn không chắc 100%, hoặc có tranh luận trong cộng đồng?
Use cases
Learning complex topics Verify AI knowledge Domain validation Documentation
Kiểm soát CL 09

Hypothesis-Driven Thinking

Đặt giả thuyết cụ thể trước khi hỏi AI, sau đó dùng AI để kiểm chứng — không phải để xác nhận. Tránh confirmation bias khi làm việc với AI.

Giải thích
Nếu bạn bắt đầu với "prove me right", AI sẽ chiều bạn. Thay vào đó: đặt hypothesis → liệt kê điều kiện để hypothesis sai → nhờ AI tìm evidence phản bác → update belief dựa trên strength of evidence, không phải cảm xúc.
Prompt mẫu
Giả thuyết của tôi: [X] 1. Điều kiện nào sẽ chứng minh giả thuyết này SAI? 2. Hãy tìm evidence mạnh nhất PHẢN BÁC giả thuyết 3. Đánh giá: evidence ủng hộ vs phản bác — cái nào mạnh hơn? 4. Kết luận được điều chỉnh là gì?
Use cases
Product decisions A/B test design Market research Technical bets
Kiểm soát CL 10

Inversion Thinking

Charlie Munger: "Hãy đảo ngược, luôn luôn đảo ngược." Thay vì hỏi làm sao để thành công, hỏi làm sao để thất bại chắc chắn. AI rất giỏi liệt kê con đường dẫn đến thất bại — hãy khai thác điều đó.

Giải thích
Thay vì "làm sao chatbot này thành công?" → "làm sao chắc chắn chatbot này thất bại?" Liệt kê tất cả con đường dẫn đến thất bại, sau đó đảo ngược từng điểm. Kết quả: một bản checklist phòng thủ mạnh hơn bất kỳ best practice nào.
Prompt mẫu
Tôi muốn [đạt mục tiêu X]. Thay vì tư vấn thành công, hãy: 1. Liệt kê 10 cách chắc chắn nhất để THẤT BẠI 2. Xếp theo mức độ nguy hiểm 3. Với mỗi cái: cơ chế thất bại là gì? Tôi sẽ tự đảo ngược danh sách đó.
Use cases
Risk management System design Strategy review Anti-patterns
Kiểm soát CL 11

Pre-mortem Analysis

Gary Klein: trước khi bắt đầu, hãy giả sử dự án đã thất bại và tìm nguyên nhân. Phát hiện rủi ro ẩn mà phân tích xuôi không bao giờ thấy.

Giải thích
Post-mortem phân tích thất bại sau khi xảy ra. Pre-mortem dùng "prospective hindsight" — giả sử thất bại đã xảy ra, hỏi ngược lại tại sao. Kỹ thuật này vượt qua confirmation bias và "planning fallacy" — hai kẻ thù lớn nhất của mọi dự án AI.
Prompt mẫu
Tưởng tượng: 6 tháng sau, dự án [X] thất bại hoàn toàn. Mọi người đang họp post-mortem. 1. 5 nguyên nhân thất bại được đề cập nhiều nhất? 2. Dấu hiệu cảnh báo nào đã bị bỏ qua từ đầu? 3. Quyết định nào là turning point tệ nhất? Bây giờ: làm gì để ngăn chặn từng điểm đó?
Use cases
Project planning Launch readiness Team alignment AI deployment
Kiểm soát CL 12

Socratic Method

Thay vì lấy câu trả lời từ AI, dùng AI để đặt câu hỏi ngược lại cho bạn. Câu hỏi hay hơn câu trả lời — vì câu hỏi đúng dẫn đến hiểu biết thật sự.

Giải thích
Socrates không dạy bằng cách đưa ra câu trả lời — ông đặt câu hỏi cho đến khi người đối diện tự phát hiện mâu thuẫn trong suy nghĩ của chính họ. Dùng AI làm "Socrates digital": ép AI hỏi bạn liên tục cho đến khi vấn đề lõi được lộ ra.
Prompt mẫu
Đừng đưa ra giải pháp. Hãy đóng vai Socrates và hỏi tôi các câu hỏi để: 1. Làm rõ giả định ẩn trong vấn đề của tôi 2. Phát hiện mâu thuẫn trong suy nghĩ của tôi 3. Dẫn tôi tự tìm ra giải pháp Hỏi từng câu một, chờ tôi trả lời.
Use cases
Problem framing Learning sessions Design thinking Root cause analysis
Kiểm soát CL 13

Critical Thinking

AI tạo ra "ảo giác năng lực" — bot nói trôi chảy không có nghĩa là bot đúng. Luôn kiểm tra giả định, nguồn gốc, độ tin cậy của mọi output.

Giải thích
Upload 20 file không có nghĩa bot hiểu domain. Bot nói trôi chảy không có nghĩa bot đúng. Test 5 câu ổn không có nghĩa production-ready. Critical thinking ép bạn kiểm: coverage đủ chưa? File có mâu thuẫn không? Câu trả lời có căn cứ không? Edge case đã test chưa?
Prompt mẫu
Với câu trả lời vừa rồi, hãy tự phản biện: 1. Giả định ẩn nào trong lập luận này? 2. Câu trả lời này là fact hay inference? 3. Bằng chứng nào ủng hộ? Bằng chứng nào phản bác? 4. Điều gì sẽ khiến câu trả lời này sai hoàn toàn?
Use cases
Validation Fact-checking Guardrails design Hallucination detection
Định hướng 14

Product Thinking

Nhìn chatbot như sản phẩm dùng thật, không phải demo. User đau ở đâu? Họ quay lại vì gì? Time-to-value có ngắn không? Ngăn bạn làm bot "ấn tượng nhưng không ai dùng."

Giải thích
Hầu hết AI demo ấn tượng trong 5 phút nhưng không ai dùng lại sau ngày đầu. Product thinking hỏi: người dùng thực sự gặp vấn đề gì? Friction nào trong flow hiện tại? Bot giải quyết đúng vấn đề đó chưa? Aha moment đến lúc nào?
Prompt mẫu
Đánh giá chatbot [X] từ góc độ product: 1. User pain point nào đang được giải quyết? 2. Time-to-value: bao nhiêu bước để user thấy giá trị? 3. Top 3 fail case phổ biến nhất là gì? 4. Tại sao user KHÔNG quay lại dùng lần 2?
Use cases
UX design Adoption analysis Retention MVP scoping
Kiểm soát CL 15

Meta Thinking

Quan sát chính cách mình đang tư duy. Mình đang framing sai bài toán chưa? Over-engineer không? Dùng prompt thay cho architecture? Từ prompt writer lên architect.

Giải thích
Meta thinking là khi bạn tự hỏi: mình đang cố giải bằng prompt cái lẽ ra phải giải bằng data structure? Mình đang nhồi instruction quá nhiều? Mình đang thiếu test set? Nên tách workflow thành các bước thay vì một prompt lớn? Đây là tư duy giúp bạn đi xa.
Prompt mẫu
Nhìn lại cách tôi đang tiếp cận vấn đề [X]: 1. Tôi đang giải ở tầng nào (prompt/data/architecture)? 2. Tầng đó có phù hợp với bản chất vấn đề không? 3. Tôi đang over-engineer hay under-engineer? 4. Nếu phải làm lại từ đầu, tôi sẽ thay đổi gì?
Use cases
Architecture review Debugging mindset Level-up thinking Self-assessment
